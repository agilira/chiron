Badge Component

Badges are inline labels used to highlight status, versions, categories, and metadata. They are lightweight, accessible, and perfect for adding visual context to your documentation.

Component Syntax

<Badge>Default</Badge>
<Badge variant="success">Success</Badge>
<Badge variant="info" size="sm">Small</Badge>
<Badge variant="error" outline>Outline</Badge>
<Badge variant="success" dot>Active</Badge>

Props

Prop Type Default Description
variant string - Color style: info, success, warning, error, primary
size string - Badge size: sm, lg (default is normal)
outline boolean false Outline style with transparent background
dot boolean false Adds colored dot indicator before text
class string - Additional CSS classes
id string - HTML id attribute
ariaLabel string - Accessibility label for screen readers
label string - Text for self-closing syntax

Accessibility

Badges are built with accessibility in mind:

  • Semantic HTML: Uses <span> elements with proper classes
  • Color Independence: Information is conveyed through text, not just color
  • Inline Display: Works naturally with text flow for screen readers
  • ARIA Support: Optional ariaLabel for additional context

Example with ARIA Label

v2.0

<Badge variant="success" ariaLabel="Version 2.0 stable release">v2.0</Badge>

Design Tokens

Badges use CSS custom properties from the Metis theme based on IBM Carbon Design System:

Variant Background Text Color
Default --color-surface-strong --color-foreground-primary
Info --color-blue-100 --color-blue-500
Success --color-green-100 --color-foreground-success-high-contrast
Warning --color-amber-100 --color-foreground-warning-high-contrast
Error --color-red-100 --color-foreground-critical-high-contrast
Primary --color-foreground-action --color-foreground-high-contrast

All badges automatically adapt to dark mode through Carbon Design System color tokens. The color palette is carefully designed to work in both light and dark themes without requiring separate styles.

Best Practices

✅ Do

  • Use badges to highlight important metadata
  • Choose appropriate variants for context (success for positive, error for negative)
  • Keep badge text short (1-3 words)
  • Use dot indicators for status (online/offline/pending)
  • Combine badges with related content (headings, lists, tables)
  • Use contrasting colors when placing badges inside callouts

❌ Don't

  • Overuse badges - they lose impact
  • Use badges for long text content
  • Rely solely on color to convey meaning
  • Use badges for interactive elements (use buttons instead)
  • Stack too many badges together
  • Use same color badge as callout background (low contrast)

Comparison with HTML

Old Way (HTML)

<span class="badge badge-success">Success</span>
<span class="badge badge-info badge-outline">Info</span>

New Way (JSX)

<Badge variant="success">Success</Badge>
<Badge variant="info" outline>Info</Badge>

Benefits:

  • Cleaner syntax: More readable and maintainable
  • Type safety: Props are validated at build time
  • Consistency: Standardized API across all components
  • Self-documenting: Clear prop names explain functionality
